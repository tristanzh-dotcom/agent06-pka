# PKA Quality Mainline Plan

Date: 2026-07-01
Status: audit draft v0.2 after external review
Owner: Agent06 PKA
Scope: theme-level knowledge aggregation, traceable answers, reusable outputs

## 1. Why This Plan Exists

Agent06 PKA has spent too much design energy on org-chart understanding. Org charts are a useful edge case, but they are not the center of a personal knowledge base. They are usually rare, slow-changing, and easy for the owner to visually inspect.

The quality mainline should return to the highest-frequency user value:

1. turn scattered personal files into coherent topic-level knowledge;
2. answer questions with visible evidence and uncertainty boundaries;
3. turn grounded answers into reusable artifacts such as interview notes, project retrospectives, English summaries, Word exports, and PPT outlines.

This document is a reviewable design and implementation plan. It is intentionally written before code changes so that TZ and the external auditor can challenge the goal, scope, ordering, and evidence model.

## 2. Current Functional Positioning

PKA is a personal knowledge-base workbench, not a general agent framework and not an org-chart analysis product.

Current strengths:

- Trusted ingestion pipeline for manual text, Office files, PDFs, images, and OCR-backed documents.
- Parse quality gates for needs-OCR, oversized chunk counts, and structured pre-chunk bypass.
- Hybrid retrieval using SQLite FTS5 and Chroma vector search, fused through RRF.
- Optional reranker with fail-open behavior.
- Dual-model generation path:
  - DeepSeek for cross-language analysis and Chinese answer generation.
  - codex-base for local English report drafting from analysis and retrieved chunks.
- Export APIs for Word and PPT.
- Source events in the SSE answer stream.

Current weaknesses:

- Retrieval returns chunks, but the system does not yet build a topic-level view of the user’s knowledge assets.
- Answer provenance is present but too coarse. Sources are listed, but claims are not explicitly mapped to evidence.
- Export is answer-centric. It does not yet support reusable output modes such as project retrospective, interview story bank, decision memo, or English executive report with structured sections.
- Knowledge quality is hard to inspect. The user cannot easily see stale topics, thin topics, ingestion failures, OCR backlog, or low-evidence answers.
- Org-chart handling is already sufficient as a specialized bypass and should not dominate the roadmap.

## 2.1 External Audit Response

The external audit identified three material design defects in v0.1. This v0.2 plan accepts those corrections.

Accepted corrections:

- `EvidenceMapper` must run after generation, not before it. PKA has no independent pre-generation fact classifier. Any claim-level mapping must inspect generated output and link it back to retrieved chunks.
- v1 evidence should be chunk-level coverage, not claim-level `verified / inferred / unsupported / advice` classification. Claim-level labels are a later capability after the output structure is stable.
- Topic aggregation requires explicit query expansion. A single `HybridRetriever.hybrid_search(query)` call is too narrow for broad requests such as "整理一下我关于 JLR 面试的所有材料".
- Knowledge Health should be brought forward as `Phase 1b`, because ingestion quality directly affects whether topic aggregation can be trusted.
- `output_mode` should not be exposed as a public API field in v1. The planner should infer modes internally until the UI and export flow are ready to support the contract.

Rejected or deferred items:

- No separate `generate_artifact` endpoint in v1. The current interaction model remains ask, inspect, export. Artifact separation can be reconsidered after structured output modes prove useful.
- No claim-level confidence labels in v1. They remain a future design target, not a first release promise.

## 3. First-Principles Design Rules

### 3.1 Facts Are Not Generated By LLMs

PKA may use LLMs for semantic understanding, summarization, translation, wording, and report drafting. It must not treat an LLM-created statement as a final source fact unless the statement is linked back to retrieved evidence.

Longer term, factual claims in a high-quality answer should fall into one of these categories:

- `verified`: directly supported by retrieved source text.
- `inferred`: derived from multiple retrieved sources with a visible reasoning bridge.
- `unsupported`: not answerable from the current knowledge base.
- `advice`: recommendation based on verified or inferred material, not a source fact.

V1 does not promise this classification. V1 only reports chunk-level evidence coverage:

- source count;
- chunk count;
- source type distribution;
- top sources used;
- low-evidence or no-evidence status.

### 3.2 Topic-Level Aggregation Before Graph Ambition

The next useful abstraction is a topic dossier, not a global knowledge graph.

A topic dossier is a deterministic runtime or cached package containing:

- topic name and aliases;
- matched source files;
- representative chunks;
- timeline hints where available;
- key entities and terms;
- evidence coverage score;
- missing-information notes;
- recommended reusable outputs.

This is enough to improve daily PKA use without building a heavy graph system.

### 3.3 Traceability Must Be User-Visible

The answer should not merely append source files. V1 should make chunk-level evidence coverage visible:

- source count;
- chunk count;
- top sources;
- source type distribution;
- query variants that did or did not retrieve evidence.

Longer term, PKA should make claim-level evidence structure visible:

- key claims;
- source chunk references;
- whether the claim is directly supported or inferred;
- confidence label;
- missing evidence, if any.

The user should be able to judge whether the answer is reliable without opening every file. V1 achieves this through coverage; later versions can add claim-to-source mapping after generation.

### 3.4 Reusable Output Is A First-Class Product Surface

For PKA, the value of an answer is often what it can become:

- interview answer;
- project retrospective;
- English report;
- work summary;
- PPT outline;
- decision memo;
- personal positioning narrative.

Reusable output modes should be structured contracts, not ad hoc prompts.

## 4. Explicit Non-Goals

This plan does not implement:

- cross-file org-chart graph fusion as a mainline feature;
- persistent global graph database;
- background LLM mining of all private files without a user-visible privacy gate;
- new external model routes;
- broad visual redesign of the shared web publishing shell;
- automatic claims that are not backed by retrieved chunks.

Org-chart code remains in maintenance mode:

- keep existing structured ingest and retrieval bias;
- fix regressions when tests fail;
- do not add a new org-chart fusion engine unless real user demand reappears.

## 5. Proposed Product Shape

### 5.1 Theme-Level Knowledge Aggregation

Add a topic aggregation layer that groups retrieved evidence into a compact dossier for the current question or selected topic.

Example user intents:

- “总结一下我关于 JLR 面试的所有材料。”
- “把 Audi 项目的经验整理成面试故事。”
- “我关于技术选型有哪些稳定观点？”
- “最近所有复盘里反复出现的问题是什么？”

Expected system behavior:

1. Build deterministic query variants from the user query.
2. Retrieve candidates for each variant with existing `HybridRetriever`.
3. Merge and deduplicate candidates by `chunk_id`.
4. Group candidates by source, source type, and topic terms.
5. Build a topic dossier from the retrieved material.
6. Pass the dossier plus selected raw chunks into generation.
7. Return answer, source list, and chunk-level coverage metadata.

### 5.1.1 Query Expansion Strategy

Add a local `QueryRewriter` before topic aggregation. It must not call an LLM in v1.

Suggested file:

- `engine/query_rewriter.py`

Input:

```json
{
  "question": "整理一下我关于 JLR 面试的所有材料"
}
```

Output:

```json
{
  "queries": [
    {"query": "JLR 面试", "reason": "base topic"},
    {"query": "JLR 面试准备", "reason": "preparation variant"},
    {"query": "JLR 职位要求", "reason": "requirement variant"},
    {"query": "JLR 面试复盘", "reason": "retrospective variant"},
    {"query": "JLR 面试反馈", "reason": "feedback variant"}
  ]
}
```

V1 variant sources:

- base noun phrases from the original query;
- project/entity token plus common personal-knowledge suffixes:
  - `准备`
  - `复盘`
  - `反馈`
  - `要求`
  - `总结`
  - `经验`
  - `面试`
  - `项目`
- English equivalents when the query contains English project names:
  - `interview preparation`
  - `interview feedback`
  - `role requirements`
  - `project retrospective`
  - `lessons learned`
- source-title hints from retrieved file names in the first-pass results.

V1 limits:

- maximum 6 query variants per user query;
- each variant retrieves at most the configured `final_top_k`;
- deduplicate by `chunk_id`;
- keep per-variant provenance for debug and audit.

### 5.2 Traceable Answer Contract

Introduce a v1 chunk-level evidence coverage contract:

```json
{
  "coverage": {
    "source_count": 4,
    "chunk_count": 9,
    "source_types": {"markdown": 3, "pdf": 1},
    "low_evidence": false,
    "query_variants": ["JLR 面试", "JLR 面试准备", "JLR 面试复盘"]
  },
  "top_sources": [
    {"source_name": "jlr_interview_notes.md", "chunk_count": 3},
    {"source_name": "jlr_role_requirements.pdf", "chunk_count": 2}
  ],
  "missing_evidence": ["No chunks were retrieved for JLR 面试反馈."]
}
```

Claim-level evidence mapping is deferred. A later version may add claim extraction after generation, but it must not be presented as verified unless the claim is linked back to retrieved chunks.

### 5.3 Reusable Output Modes

Add output templates as explicit modes:

- `answer`: normal grounded answer.
- `retrospective`: project replay with background, actions, results, lessons.
- `interview_story`: STAR-style interview narrative with evidence anchors.
- `english_report`: executive summary, key findings, implications, sources.
- `ppt_outline`: slide-level outline with title, point, evidence, and speaker note.
- `decision_memo`: context, options, tradeoffs, recommendation, risks.

The modes should use the same evidence contract. Only the presentation structure changes.

## 6. Target Architecture

```text
User query
  -> QueryRewriter
  -> HybridRetriever
  -> TopicAggregator
  -> AnswerPlanner
  -> Generator
  -> EvidenceMapper
  -> SSE answer + sources + chunk-level coverage metadata
  -> Export adapters
```

### 6.1 `QueryRewriter`

Responsibility:

- Generate deterministic query variants for broad personal-knowledge requests.
- Keep the original query as the first and highest-priority query.
- Explain why each variant was produced.
- Avoid LLM calls in v1.

Suggested file:

- `engine/query_rewriter.py`

Suggested dataclasses:

- `QueryVariant`
- `QueryExpansionPlan`

### 6.2 `TopicAggregator`

Responsibility:

- Consume multi-query retrieval results.
- Deduplicate chunks by `chunk_id`.
- Group retrieved chunks by topic terms, source, source type, date hints when available, and query variant.
- Produce a compact topic dossier.
- Avoid generating facts. It only organizes retrieved material.

Suggested file:

- `engine/topic_aggregator.py`

Suggested dataclasses:

- `TopicDossier`
- `TopicEvidenceGroup`
- `TopicCoverage`

### 6.3 `AnswerPlanner`

Responsibility:

- Choose an internal output mode.
- Select prompt framing.
- Decide whether to use normal answer, retrospective, interview story, report, PPT outline, or decision memo.

Suggested file:

- `engine/answer_planner.py`

First version should be conservative:

- no public `output_mode` API field;
- infer mode only from the query and language;
- default to `answer`;
- expose inferred mode only in debug/trace metadata.

### 6.4 Generator Integration

Responsibility:

- Preserve current DeepSeek/codex-base route.
- Add topic dossier context when available.
- Keep fallback behavior when JSON is partial or malformed.

Suggested files:

- modify `engine/generator.py`
- modify `server.py`

### 6.5 `EvidenceMapper`

Responsibility:

- Run after generation.
- Build chunk-level coverage from retrieved chunks and query variants.
- Attach coverage metadata to SSE source/debug events.
- Avoid claim-level `verified / inferred / unsupported / advice` labels in v1.

Suggested file:

- `engine/evidence.py`

First version should be conservative:

- it maps chunks, not claims;
- it summarizes evidence coverage;
- it can note missing query variants;
- it never invents evidence or quotes.

### 6.6 Export Integration

Responsibility:

- Export structured outputs, not only raw answer text.
- Word export should include evidence table.
- PPT outline should include slide sections and source anchors.

Suggested files:

- modify `engine/exporter.py`
- modify `engine/ppt_maker_adapter.py` only if higher-quality PPT generation needs the new structure
- modify `server.py` export request schema if structured export is exposed

## 7. API Contract Changes

Do not expose `output_mode` in v1.

Extend `POST /api/query` request only with optional trace behavior:

```json
{
  "question": "把 Audi 项目经验整理成面试故事",
  "language": "zh",
  "debug": false,
  "trace": true
}
```

Fields:

- `trace`: optional; when true, return chunk-level coverage metadata in SSE sources/debug event.
- `output_mode`: not public in v1; mode inference is internal and appears only in debug/trace metadata.

SSE event extension:

```json
{
  "type": "sources",
  "source_status": "grounded",
  "sources": [],
  "evidence": {
    "coverage": {
      "source_count": 4,
      "chunk_count": 9,
      "low_evidence": false,
      "query_variants": ["Audi 项目", "Audi 项目复盘", "Audi 技术选型"]
    },
    "top_sources": [],
    "missing_evidence": []
  }
}
```

Backward compatibility:

- Existing clients that ignore `evidence` continue working.
- Existing `language` behavior remains unchanged.
- Existing source payload fields remain unchanged.

## 8. Implementation Plan

### Phase 0: Audit Baseline

Goal:

- Freeze the current behavior before adding quality layers.

Actions:

1. Run focused tests for retrieval, generation, export, and async OCR task handling.
2. Record current API payload shape for `/api/query` and export endpoints.
3. Confirm no model route changes are needed.

Suggested verification:

```bash
python3 -m pytest -q tests/test_indexer_retriever.py tests/test_generator_api.py tests/test_export_api.py tests/test_async_ocr_pipeline.py
```

Expected result:

- Existing tests pass before feature implementation begins.

### Phase 1: Topic Dossier Core

Goal:

- Add deterministic query expansion and topic aggregation without changing generation behavior.

Files:

- create `engine/query_rewriter.py`
- create `engine/topic_aggregator.py`
- create `tests/test_query_rewriter.py`
- create `tests/test_topic_aggregator.py`

Core behaviors:

- generate up to 6 deterministic query variants;
- preserve variant reason and provenance;
- run multi-query retrieval through the existing retriever integration point;
- deduplicate chunks by `chunk_id`;
- group chunks by normalized source name;
- extract topic terms from question using existing meaningful-term logic or a local equivalent;
- calculate coverage metrics:
  - source count;
  - chunk count;
  - source type mix;
  - top source names;
  - low-evidence flag.
- produce Markdown context that can be passed to the generator later.

Acceptance criteria:

- Query expansion for "JLR 面试" includes preparation, requirements, retrospective, and feedback variants.
- Query expansion does not call any LLM.
- Multi-query retrieval deduplicates repeated chunks.
- Empty chunk list returns an empty dossier with `low_evidence=true`.
- Multiple chunks from the same source are grouped.
- The aggregator does not call any LLM.
- Org-chart chunks are treated as normal evidence groups, not as a special roadmap driver.

### Phase 1b: Simple Knowledge Health Endpoint

Goal:

- Add an early deterministic health surface so topic aggregation quality can be judged against corpus quality.

Files:

- modify `server.py`
- create or extend tests around the health endpoint

Core behaviors:

- expose `/api/knowledge/health`;
- return indexed source count, total chunk count, source type distribution, OCR task status counts, and recent task quality actions where available;
- do not call LLMs;
- do not scan raw private files recursively.

Acceptance criteria:

- Endpoint returns deterministic JSON.
- Endpoint works when there are no OCR task files.
- Endpoint does not require DeepSeek, codex-base, reranker, or Ollama.

### Phase 2: Answer Mode Planner

Goal:

- Introduce internal reusable output mode inference while keeping the public API unchanged.

Files:

- create `engine/answer_planner.py`
- create `tests/test_answer_planner.py`
- modify `server.py`

Core behaviors:

- infer answer mode from query and language;
- keep mode internal in v1;
- pass the selected mode into generation.

Acceptance criteria:

- Existing `/api/query` request schema remains backward-compatible.
- "整理成面试故事" infers `interview_story`.
- "生成英文报告" or `language=en` infers `english_report`.
- Unknown or generic questions infer `answer`.

### Phase 3: EvidenceMapper v1

Goal:

- Create a post-generation chunk-level coverage structure that can be returned in debug/trace mode.

Files:

- create `engine/evidence.py`
- create `tests/test_evidence.py`
- modify `engine/generator.py` only where needed to attach metadata after generation

Core behaviors:

- define `EvidenceCoverage`, `EvidenceSourceSummary`, and `EvidenceReport`;
- map source metadata from retrieved chunks into source summaries;
- label coverage as `grounded`, `thin`, or `no_answer`;
- list query variants that produced no chunks;
- do not classify individual claims in v1.

Acceptance criteria:

- Evidence mapping works without LLM calls.
- No-answer responses return `coverage_status=no_answer`.
- Low chunk/source counts return `coverage_status=thin`.
- The report contains source count, chunk count, top sources, source type distribution, and missing query variants.
- Existing source events remain backward-compatible.

### Phase 4: Generator Prompt Upgrade And Export Upgrade

Goal:

- Make generation use the topic dossier and internal output mode while preserving current fallbacks, then upgrade exports after the generated structure is stable.

Files:

- modify `engine/generator.py`
- modify `engine/exporter.py`
- extend `tests/test_generator_api.py`
- extend `tests/test_export_api.py`

Core behaviors:

- accept optional `topic_dossier` and internal `answer_mode` parameters;
- include topic dossier context in DeepSeek prompts;
- ask DeepSeek for structured `key_facts`, `logic_chain`, and `missing_evidence`;
- keep partial JSON recovery path;
- for `english_report`, preserve current DeepSeek-to-codex-base path;
- for `ppt_outline`, produce slide-structured text first, not binary PPT directly.
- Word export may include coverage table after `EvidenceReport` is stable.

Acceptance criteria:

- Default answer mode still behaves like current PKA.
- `interview_story` returns a structured narrative with source anchors.
- `retrospective` returns situation, actions, outcome, lessons, and evidence.
- `ppt_outline` returns slide titles and evidence notes.
- DeepSeek failure still returns an SSE error event and sources, not a server crash.

### Phase 5: Full Knowledge Health Frontend Surface

Goal:

- Surface the health endpoint in the UI after the backend contract is stable.

Files:

- optional UI changes in `static/ask.html`, `static/app.js`, and `static/style.css`
- relevant frontend tests if present

Core behaviors:

- show source/chunk counts, OCR queue status, and recent quality actions;
- keep it read-only;
- do not add broad web publishing shell changes.

Acceptance criteria:

- Existing clients remain compatible.
- UI reads `/api/knowledge/health` without triggering ingestion or LLM work.
- Text and controls fit inside existing Agent06 local UI boundaries.

## 10. Privacy and Model Routing

No new model route is proposed.

Existing route usage:

- `local_semantic_retrieval`: Ollama `bge-m3`, local only, for embeddings and vector retrieval.
- `cross_language_analysis_and_zh_answer`: DeepSeek, sends question text and retrieved chunks when configured.
- `english_report_generation`: codex-base local runtime route, receives DeepSeek analysis and retrieved chunks for English report drafting.

Privacy implications:

- Query expansion, topic aggregation, knowledge health, and v1 evidence mapping must run locally and deterministically.
- If DeepSeek is enabled, retrieved chunks are already sent under the existing Agent06 route. This plan does not expand the category of data sent.
- If future background topic indexing is proposed, it must get a separate privacy review before implementation.

## 10.1 Short Release Slice

If the first implementation must be small, implement only:

1. `QueryRewriter`;
2. `TopicAggregator`;
3. simple `/api/knowledge/health`;
4. internal `AnswerPlanner`;
5. optional chunk-level coverage in trace/debug metadata.

Defer:

- generator prompt upgrades;
- export upgrades;
- claim-level evidence mapping;
- UI health surface.

## 11. Audit Questions For External Review

The auditor should challenge these points directly:

1. Is `TopicDossier` the right next abstraction, or should PKA first improve ingestion health visibility?
2. Is claim-level evidence mapping feasible with current model output, or should v1 only expose chunk-level evidence coverage?
3. Should `output_mode` be a user-facing API field now, or should it remain an internal planner decision until UI is ready?
4. Does `trace=true` create too much payload for SSE clients?
5. Should reusable outputs be generated directly by `generate_answer`, or should there be a separate `generate_artifact` endpoint?
6. Are the proposed confidence labels precise enough for personal knowledge work?
7. Does any part of this plan accidentally expand external model privacy exposure?
8. Which phase should be cut if the first implementation must fit into a short release window?

## 12. Recommended First Implementation Slice

The safest first slice is:

1. `QueryRewriter`;
2. `TopicAggregator`;
3. simple `/api/knowledge/health`;
4. internal `AnswerPlanner`;
5. chunk-level `EvidenceReport` only in trace/debug metadata;
6. no public `output_mode` field;
7. no UI changes except preserving existing behavior.

This gives PKA better structure without destabilizing generation, export, or the visible workbench. Once this slice is reviewed and tested, reusable output modes can be added one by one.

## 13. Review Protocol

After the external audit returns:

1. classify each audit comment as `accept`, `reject`, or `needs clarification`;
2. for accepted comments, update this plan before implementation;
3. for rejected comments, document the technical reason;
4. if the audit changes model routing, stop and update the global model routing record with TZ approval;
5. only after the plan is approved, write TDD tests and begin implementation.
