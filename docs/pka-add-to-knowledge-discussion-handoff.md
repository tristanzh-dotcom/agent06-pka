# PKA Add-To-Knowledge Discussion Handoff

Date: 2026-07-03
Status: design handoff draft
Scope: generated answer "加入知识库" capability

## 1. Purpose

This document captures the current discussion about adding model-generated answers back into PKA's personal knowledge base.

The next conversation should use this as the starting point for designing and implementing the "加入知识库" function. It should not restart the debate around export or generated asset library unless those topics directly affect the add-to-knowledge design.

## 2. Current Understanding Of PKA

PKA is currently a RAG-style personal knowledge-base system.

Current question flow:

1. User asks a question.
2. PKA retrieves relevant chunks from the local personal knowledge base.
3. PKA sends the original user question plus retrieved chunks to the configured model route.
4. The model generates an answer.
5. The answer is streamed back to the UI.
6. The answer can be exported manually, but it is not automatically persisted as a new knowledge source.

Important current behavior:

- The user's original question is preserved in the LLM prompt.
- Query expansion may be used for retrieval, but it does not replace the original question.
- Recent input-fidelity work adds adjacent same-source chunks before generation so the model sees more continuous source context.
- Model output is currently a mostly one-time result unless exported.

## 3. Problem Statement

High-quality model outputs are valuable personal assets.

When PKA spends tokens to synthesize a user's own materials into a project retrospective, interview answer, English report, decision memo, or technical judgment, that output can become useful future knowledge.

Current gap:

- generated answers are not persisted by default;
- generated answers are not searchable unless manually copied back into the system;
- there is no structured record of the relationship between question, answer, evidence chunks, model route, and timestamp;
- valuable second-order knowledge evaporates after the session unless exported.

The target function is specifically:

> Allow the user to deliberately add a generated answer into the RAG knowledge base, while preserving source provenance and preventing model-generated text from being mistaken for primary evidence.

## 4. Terminology Boundaries

The discussion separated three actions:

### 4.1 Export

Export is a one-time action that turns the current answer into a document artifact such as Word or PPT.

Export does not imply:

- future searchability;
- RAG ingestion;
- asset management;
- long-term replay in the app.

### 4.2 Save As Asset

Save as asset means storing the generated answer in a generated asset library.

This is larger than add-to-knowledge because assets should later support:

- browsing;
- review;
- reuse;
- re-editing;
- flowing back into chat;
- optionally being promoted into the knowledge base.

This requires a real UI surface and should be treated as a separate product feature.

### 4.3 Add To Knowledge Base

Add to knowledge base means the generated answer becomes searchable in future RAG retrieval.

This is the current focus.

It is smaller than a full asset library because it can be implemented as a deliberate one-shot action:

1. user reviews answer;
2. user clicks "加入知识库";
3. system creates a generated-source record;
4. system indexes it with strict metadata;
5. future retrieval can find it, with clear generated-source labeling.

## 5. First-Principles Requirement

Model output must not be treated as a primary fact source.

The global model routing rule says facts are not generated by language models. PKA must preserve that boundary.

Therefore, "加入知识库" must not simply insert the model answer as if it were a normal uploaded note.

It must mark the inserted content as derived, generated, and evidence-linked.

Required metadata:

```json
{
  "source_type": "generated_asset",
  "generated": true,
  "not_primary_source": true,
  "derived_from_chunk_ids": ["source.md#0", "source.pdf#3"],
  "derived_from_sources": ["source.md", "source.pdf"],
  "question": "original user question",
  "language": "zh|en",
  "model_route": "deepseek|codex-base|dual",
  "created_at": "ISO timestamp",
  "evidence_coverage_status": "grounded|thin|no_answer",
  "user_confirmed_for_knowledge_base": true
}
```

## 6. Expected RAG Impact

If an answer is added to the knowledge base, future questions may retrieve it.

This is desirable when:

- the generated answer is a useful synthesis;
- it reflects the user's confirmed current view;
- it summarizes scattered source materials into a reusable narrative;
- it can serve as a second-order personal knowledge artifact.

This is dangerous when:

- the answer contains unsupported claims;
- the answer is based on thin retrieval coverage;
- the answer is model hallucination;
- future RAG treats it as stronger than primary source material;
- repeated generated answers recursively amplify each other.

The add-to-knowledge implementation must therefore control retrieval influence.

## 7. Retrieval Policy For Generated Knowledge

Generated knowledge should be searchable, but it should not silently dominate primary evidence.

Recommended v1 policy:

- generated entries are indexed into both FTS and vector search;
- generated entries use `source_type="generated_asset"`;
- retriever keeps the `source_type` visible in returned chunks;
- generator prompt labels generated chunks clearly;
- when both primary and generated chunks are retrieved, primary source chunks should be preferred for factual claims;
- generated chunks may be used as synthesis, framing, or prior personal viewpoint;
- no recursive automatic re-ingestion of generated answers.

Possible later policy:

- give generated chunks a slight retrieval penalty unless the query asks for "我的总结", "之前的判断", "我上次怎么说";
- boost generated chunks when the user explicitly asks for prior conclusions, summaries, narratives, or generated reports;
- expose a filter to include or exclude generated knowledge.

## 8. Generation Prompt Requirement

When future RAG retrieves generated knowledge, the model must be told what it is reading.

Prompt should distinguish:

```text
[Primary source chunks]
These are user-uploaded or user-entered source materials.

[Generated knowledge chunks]
These are previous model-generated syntheses saved by the user into the knowledge base.
They may express the user's confirmed prior synthesis, but they are not primary evidence.
Use them as secondary context and prefer primary source chunks for factual claims.
```

This is necessary because otherwise generated summaries can become self-reinforcing source material.

## 9. Suggested V1 Data Model

V1 can avoid building a full generated asset library.

Recommended v1 storage shape:

- create a generated Markdown source file under the configured PKA data directory, for example:

```text
{data_dir}/generated/knowledge/YYYY-MM-DD/generated_answer_<id>.md
```

File content:

```markdown
# Generated Knowledge Entry

Question:
...

Answer:
...

Sources:
- source.md#0
- source.pdf#3

Evidence Coverage:
- status: grounded
- source_count: 3
- chunk_count: 8

Model Route:
- language: zh
- route: deepseek

Generated At:
2026-07-03T...

---

[GENERATED_KNOWLEDGE]
...
[/GENERATED_KNOWLEDGE]
```

Then ingest this file through the same internal indexing path used for text/manual input, with metadata marking it as generated.

Alternative:

- skip writing a Markdown file and directly insert a `Chunk` with `source_type="generated_asset"`.

Recommendation:

- write a physical Markdown file first.

Reason:

- easier audit;
- easier backup;
- easier user inspection;
- avoids invisible database-only generated knowledge;
- aligns with personal knowledge base ownership.

## 10. Suggested API Contract

Add a new endpoint:

```http
POST /api/knowledge/add-generated
```

Request:

```json
{
  "question": "...",
  "answer": "...",
  "sources": [
    {
      "source_name": "source.md",
      "source_type": "md",
      "chunk_index": 0,
      "chunk_id": "source.md#0",
      "relevance": 0.91,
      "raw_file_path": ""
    }
  ],
  "evidence": {
    "coverage": {
      "coverage_status": "grounded",
      "source_count": 2,
      "chunk_count": 5
    }
  },
  "language": "zh",
  "answer_mode": "answer|interview_story|retrospective|english_report|ppt_outline|decision_memo"
}
```

Response:

```json
{
  "status": "ok",
  "source_name": "generated_answer_20260703_153012.md",
  "source_type": "generated_asset",
  "chunks": 1,
  "chunk_ids": ["generated_answer_20260703_153012.md#0"],
  "raw_file_path": "generated/knowledge/2026-07-03/generated_answer_20260703_153012.md"
}
```

## 11. Acceptance Criteria

V1 should satisfy:

- user-triggered only; no automatic write-back;
- writes a physical generated Markdown file;
- indexes the generated answer into RAG;
- marks generated content with `source_type="generated_asset"`;
- stores provenance to original question and source chunks;
- future `/api/query` can retrieve generated entries;
- generated entries are visible as generated in source metadata;
- adding generated content must not call an LLM;
- generated entries must not be treated as primary source facts;
- no recursive auto-ingest of generated answers.

## 12. Tests To Write First

Recommended TDD tests:

1. API creates a generated Markdown file under `data_dir/generated/knowledge/YYYY-MM-DD/`.
2. API indexes generated content and returns `source_type="generated_asset"`.
3. Metadata includes original question, source chunk IDs, evidence coverage, language, answer mode, and generated flag.
4. Query can retrieve a generated entry after add-to-knowledge.
5. Generator prompt labels generated chunks as generated/secondary when source_type is `generated_asset`.
6. API rejects empty answer or empty question.
7. API does not call DeepSeek/codex-base.
8. Adding generated knowledge twice creates separate deterministic files or deduplicates by content hash. This policy must be chosen before implementation.

## 13. Open Design Decisions

The next conversation should resolve these before coding:

1. Should duplicate generated answers be allowed, or deduplicated by hash?
2. Should generated entries be indexed as one chunk or chunked normally?
3. Should generated entries have a retrieval penalty in v1?
4. Should "加入知识库" be blocked when evidence coverage is `no_answer` or `thin`?
5. Should generated entries be editable before indexing?
6. Should generated Markdown include full source excerpts or only source references?
7. Should the endpoint reuse `_ingest_parsed_result()` or call `runtime.indexer.upsert()` directly?

Recommended defaults:

- allow generated entries only when question and answer are non-empty;
- warn but do not block `thin` evidence in backend v1;
- block `no_answer` entries unless user explicitly forces it later;
- write one generated Markdown file per save action;
- chunk normally if the answer is long, but preserve generated metadata on all chunks;
- reuse `_ingest_parsed_result()` if metadata can be preserved; otherwise create a narrow helper for generated knowledge ingestion.

## 14. Explicit Non-Goals For The Next Conversation

Do not implement full generated asset library UI in this slice.

Do not build:

- asset browsing;
- asset editing;
- asset collections;
- asset search UI;
- asset-to-chat replay;
- external GitHub skill/software integration;
- automatic background answer capture.

Those belong to a later generated asset library product discussion.

The immediate target is only:

> User-reviewed generated answer can be deliberately added into RAG knowledge base with provenance and generated-source safeguards.
